# Чек-лист соблюдения методики Киры

**Проект:** 1C Log Parser Service  
**Применение:** После каждого обсуждения/изменения требований

---

## Критерии проверки

### 1. Работа с требованиями (Requirements)

- [ ] **REQ-1:** Любое новое требование зафиксировано в разделе Requirements спеки
- [ ] **REQ-2:** Требование обсуждено и явно утверждено пользователем
- [ ] **REQ-3:** Источник требования указан (ссылка на документ, обсуждение)
- [ ] **REQ-4:** Переход к дизайну происходит только после утверждения требования

### 2. Работа с дизайном (Design)

- [ ] **DES-1:** Изменения архитектуры зафиксированы в разделе Design
- [ ] **DES-2:** Проанализировано влияние изменения на Requirements
- [ ] **DES-3:** Обновлены схемы/диаграммы (если применимо)
- [ ] **DES-4:** Дизайн-решение утверждено пользователем перед реализацией

### 3. Работа с задачами (Implementation Tasks)

- [ ] **TASK-1:** Задачи создаются только на основе утверждённого Design
- [ ] **TASK-2:** У каждой задачи есть чёткий acceptance criteria
- [ ] **TASK-3:** Статус задач актуален (pending/in_progress/completed/cancelled)
- [ ] **TASK-4:** Завершённые задачи помечены датой завершения

### 4. Документация и ссылки

- [ ] **DOC-1:** Все ссылки на файлы актуальны (#[[file:...]])
- [ ] **DOC-2:** Changelog обновлён (версия, дата, описание изменений)
- [ ] **DOC-3:** Зависимости и источники документированы
- [ ] **DOC-4:** Изменения спеки прослеживаемы (git commits с описанием)

### 5. Качество процесса

- [ ] **PROC-1:** Не пропущены этапы (Requirements → Design → Tasks → Implementation)
- [ ] **PROC-2:** Обратная связь от пользователя получена перед продолжением
- [ ] **PROC-3:** Нет "прыжков" в реализацию без утверждённого дизайна
- [ ] **PROC-4:** Спека остаётся единым источником истины (Single Source of Truth)

---

## Формат отчёта

После каждого значимого обсуждения заполняем отчёт:

```
✅ Чек-лист Киры: X/20

Раздел                  | Статус
------------------------|--------
Требования (REQ)        | ✓ 4/4
Дизайн (DES)            | ✓ 4/4
Задачи (TASK)           | ✓ 4/4
Документация (DOC)      | ✓ 4/4
Процесс (PROC)          | ✓ 4/4

Замечания:
- [Если есть несоответствия, описываем здесь]
- [План по устранению]

Действия:
- [ ] [Конкретные шаги для исправления, если чек-лист не пройден]
```

---

## Правила применения

1. **Обязательная проверка:**
   - После добавления нового требования
   - После изменения архитектуры/дизайна
   - Перед началом нового sprint
   - Перед мержем крупных изменений

2. **Блокирующие условия:**
   - Если REQ не пройдены → СТОП, фиксируем требования
   - Если DES не пройдены → СТОП, доводим дизайн
   - Если PROC-3 нарушен → СТОП, возвращаемся к утверждению дизайна

3. **Непрерывное улучшение:**
   - Если чек-лист регулярно не проходит → пересмотреть процесс
   - Если какие-то пункты неприменимы → обсудить с пользователем и адаптировать
